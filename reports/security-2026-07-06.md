## Findings

### [SEC-001] CLI generic exception fallback prints raw exception text to console — LOW
**Category**: Sensitive data exposure (local-only)
**Location**: `main.py:154-157`, `196-199`, `245-248` (pre-existing code, unmodified by this diff, but now sees 3x the exception surface)
**Reproduction**: Any exception raised during an LLM call that does **not** match one of the enumerated types (`PermissionDenied`, `GoogleAPIError`, `openai.OpenAIError`, `anthropic.AnthropicError`, `ConnectionError`/`ollama.ResponseError`) falls to `except Exception as exc: console.print(f"[red]Error:[/] Unexpected error: {exc}")`. If such an exception's `str()` happens to embed key material (some upstream auth-failure error text echoes back the offending key, e.g. `"Incorrect API key provided: sk-..."`), it would be printed verbatim to the user's own terminal.
**Impact**: Low — this is the CLI, so the only "leak" is to the same person who owns the key in their own `.env`, not to a network peer or log aggregator. Verified the new provider-specific branches (`openai.OpenAIError`, `anthropic.AnthropicError`, ollama branch) correctly route `exc` only to `logging.error(...)` and print a static, non-interpolated message to the console — so the leak path requires an SDK exception type outside the enumerated hierarchy. Checked `anthropic.OpenAIError`/`anthropic.AnthropicError` MRO directly (`Exception` is the sole parent) — all SDK-specific errors (including auth failures) are subclasses, so this path is narrow, not a real bypass, and was not introduced by this diff.
**Fix**: Change the CLI's final catch-all in all three sites to print a static message (`"Unexpected error — see logs."`) and keep `exc` only in `logging.error`, mirroring the pattern already used correctly by the new provider-specific branches directly above it.

### [SEC-002] `ConnectionError`/`ollama.ResponseError` except clause is not provider-scoped — LOW (confirms implementer's label, doesn't just accept it)
**Category**: Code quality / log accuracy (verified no security consequence)
**Location**: `main.py:150,192,241`; `api/main.py:154,207`
**Reproduction**: With `LLM_PROVIDER=openai` (or `anthropic`), a raw builtin `ConnectionError` originating from anywhere else inside the same `try` block (e.g. a socket-level failure in the RAG retrieval step, which runs inside the same `chain.invoke()` call, before the SDK gets a chance to wrap it) would be caught by this branch and logged/displayed as an "Ollama error" even though Ollama is not the active provider.
**Impact**: Verified this is cosmetic, not a security or correctness-of-response issue: in `api/main.py` every one of these branches (openai/anthropic/ollama) produces the **identical** generic client-facing string (`"Upstream LLM error — the model provider rejected the request."`, HTTP 502) — misattribution only affects the server-side log label, not what the client sees, and doesn't suppress a more specific/actionable path since none of the sibling branches are more actionable than this one. In the CLI, the console message does differ per branch ("Upstream LLM error (Ollama)." vs "(OpenAI)."), so a user on `LLM_PROVIDER=openai` hitting this path would see a misleading provider name — a UX/triage nit, not an exploitable condition. Confirms the implementing agent's LOW/cosmetic classification was accurate; no escalation warranted.
**Fix** (optional, non-blocking): scope the `ConnectionError` catch to only the Ollama branch by checking the active provider, or rename the log line to "Connection error" without a provider name when the source is ambiguous.

## Checks performed with no findings (explicitly verified, not assumed)

1. **API key handling across all 4 providers**: No key is ever logged, printed, or echoed into a client/console-facing message. Every new `except <provider>.<X>Error as exc:` branch in `main.py` and `api/main.py` routes `exc` only to `logging.error`/`logger.error` (server-side); the message shown to the console/HTTP client is a static string in every case. Confirmed via direct read of all `_invoke_chain_safely`, `_run_agent_step_safely`, `chat_with_llm`, `run_agent_turn` bodies.
2. **Fail-open vs fail-closed**: An invalid `LLM_PROVIDER` always calls `_fail()` → `sys.exit(1)`; the trailing `return None` statements in `llm_provider.py` are unreachable in practice. Confirmed `api/main.py`'s `lifespan()` calls `get_llm()` with **no** surrounding `try/except` (not even a bare `except Exception`, which wouldn't catch `SystemExit` anyway since it inherits `BaseException`) — a fatal misconfiguration correctly aborts FastAPI startup entirely, no `llm=None` code path exists.
3. **No client-controlled provider/model override**: `api/models.py` request schemas (`ChatRequest`, `AgentRequest`, etc.) expose no `model`/`provider` field — a remote API client cannot influence provider or model selection per-request; it is fixed at process startup from server-side env only.
4. **Env var parsing**: both `LLM_PROVIDER` and each API key go through `.strip()`; provider matching is `.lower()`'d, so whitespace-only or mixed-case values can't produce a false "configured" state or bypass the intended provider list.
5. **Secrets hygiene**: `.env` is gitignored and untracked; `git grep` across tracked files for common key prefixes (`sk-ant-`, `sk-proj-`, generic `sk-...`) found nothing committed.
6. **Startup-vs-runtime validation gap**: confirmed sufficiently handled, not a gap — an externally revoked key mid-process on the long-running API surfaces through the new `openai.OpenAIError`/`anthropic.AnthropicError`/`PermissionDenied` branches already added, producing a clean 502 rather than a raw exception.
